  791.  check.CONNECTION_REFUSED     connection to a port was refused 
  792.  check.SSL_HANDSHAKE_FAILURE  SSL connect problem, most likely untrusted certificate
  793.  check.COMMUNICATION_FAILURE  generic communication failure

  795.  check.AUTH_NOT_SUPPORTED     authentication flavor not supported. LDAP server probably 
  796.                               configured to not allow passwords
  797.  check.NAME_NOT_FOUND         unable to resolve an LDAP name. most likely invalid search base

  1691. WaitSet: scalable mechanism for listening for changes to one or more accounts 
  1692.  
  1693. INTEREST_TYPES: comma-separated list.  Start with:
  1694.    c: contacts
  1695.    m: msgs (and subclasses)
  1696.    a: all
  1697.  
  1698.  
  1699. *************************************
  1700. CreateWaitSet: must be called once to initialize the WaitSet
  1701. and to set its "default interest types"
  1702. ************************************* 
  1703. <CreateWaitSetRequest defTypes="DEFAULT INTEREST_TYPES">
  1704.   [ <add>
  1705.       [<a id="ACCTID" [token="lastKnownSyncToken"] [types="if_not_default"]/>]+
  1706.     </add> ]
  1707. </CreateWaitSetRequest>
  1708.  
  1709. <CreateWaitSetResponse waitSet="setId" defTypes="types" seq="0">
  1710.   [ <error ...something.../>]*
  1711. </CreateWaitSetResponse>  
  1712.  
  1713.  
  1714. ************************************
  1715. WaitMultipleAccounts:  optionally modifies the wait set and checks
  1716. for any notifications.  If block=1 and there are no notificatins, then
  1717. this API will BLOCK until there is data.
  1718.  
  1719. Client should always set 'seq' to be the highest known value it has
  1720. received from the server.  The server will use this information to
  1721. retransmit lost data.
  1722.  
  1723. If the client sends a last known sync token then the notification is
  1724. calculated by comparing the accounts current token with the client''s
  1725. last known.
  1726.  
  1727. If the client does not send a last known sync token, then notification
  1728. is based on change since last Wait (or change since <add> if this
  1729. is the first time Wait has been called with the account)
  1730. *************************************
  1731. <WaitMultipleAccountsRequest waitSet="setId" seq="highestSeqKnown" [block="1"]>
  1732.   [ <add>
  1733.       [<a id="ACCTID" [token="lastKnownSyncToken"] [types]/>]+
  1734.     </add> ]
  1735.   [ <update>
  1736.       [<a id="ACCTID" [token="lastKnownSyncToken"] [types]/>]+
  1737.     </update> ]  
  1738.   [ <remove>
  1739.       [<a id="ACCTID"/>]+
  1740.     </remove> ]  
  1741. </WaitMultipleAccountsRequest>
  1742.  
  1743. <WaitMultipleAccountsResponse waitSet="setId" [seq="seqNo" OR canceled="1"]>
  1744.   [ <n id="ACCTID"/>]*
  1745.   [ <error ...something.../>]*
  1746. </WaitMultipleAccountsResponse>
  1747.  
  1748. If the specified wait set does not exist, the server will throw an
  1749. admin.NO_SUCH_WAITSET exception.
  1750.  
  1751. NOTE: an empty response to a blocking request *is* possible: it would
  1752. happen if the server timed-out the waiting.  The server does this
  1753. occasionally just so that requests don''t get "stuck".  The client
  1754. should re-submit the original request if this happens.
  1755.  
  1756. If a second WaitMultiple request arrives at the server while one is
  1757. already waiting, the first request will be immediately completed and
